
## Общая оценка

Главная проблема не в том, что архитектура «плохая», а в том, что она сейчас **не определена окончательно**. В репозитории одновременно существуют:

* FastAPI-приложение с Jinja-шаблонами и JavaScript;
* отдельная фабрика Dash-приложения;
* Dash callbacks;
* прикладные use cases;
* файловое хранилище сессий;
* вычислительный код, тесно связанный с pandas и Plotly.

Получился переходный монолит, разбитый по папкам, но не по реальным границам ответственности.

## Архитектурные проблемы

### 1. Одновременно существуют две конкурирующие web-архитектуры

Основная точка запуска создаёт FastAPI-приложение через `tgapp.web.app.create_app`, но рядом существует отдельный `app_factory.py`, создающий Dash-приложение, а также набор Dash callbacks и собственный layout. Это создаёт два способа обрабатывать UI, состояние, загрузки и графики. ([GitHub][2])

Последствия:

* исправления приходится потенциально делать дважды;
* непонятно, какой путь является основным;
* части старого Dash-кода могут незаметно расходиться с FastAPI-реализацией;
* разработчик не может определить архитектурные правила по существующему коду.

Нужно выбрать один вариант. Для этого приложения я бы оставил **FastAPI + Jinja/HTMX или небольшой JavaScript**, а Dash-код удалил либо вынес в отдельную экспериментальную ветку.

### 2. Слои существуют только на уровне названий каталогов

`application/use_cases.py` напрямую импортирует:

* pandas;
* файловые парсеры;
* Plotly-подготовку;
* архиватор;
* конкретный `SessionStorage`.

При этом модуль инфраструктуры построения графиков импортирует application DTO и функции доменного слоя, а view model напрямую обращается и к use cases, и к инфраструктурной сериализации. Направление зависимостей получается циклическим по смыслу: `application → infrastructure`, `infrastructure → application`, `web → всё сразу`. ([GitHub][3])

То есть сейчас слои не дают основных преимуществ:

* вычислительную логику трудно тестировать отдельно;
* файловое хранение нельзя заменить без изменения use case;
* Plotly становится частью прикладной логики;
* web-представление знает внутреннюю структуру хранилища.

### 3. Нет чёткой модели предметной области

Доменные модели содержат `pandas.DataFrame`, поля `Any`, строковые статусы и слабо валидируемые dataclass-настройки. DTO также в основном представлены как `dict[str, Any]`. ([GitHub][4])

Для приложения обработки термограмм стоило бы явно определить:

* физические величины и их единицы;
* тип оси: время или температура;
* требования к монотонности;
* обязательные и опциональные колонки;
* допустимые диапазоны параметров;
* ошибки валидации;
* различие между исходной, нормализованной и обработанной термограммой.

Сейчас почти любой DataFrame может пройти глубоко в вычислительный pipeline, а ошибка будет либо замаскирована, либо проявится уже при построении графика.

### 4. Состояние сессии одновременно является БД, файловой системой и моделью UI

Сессия хранится в каталоге, а её состояние распределено между JSON, CSV и другими файлами. Web-слой получает путь `session_dir`, use cases знают расположение файлов, а загрузка страницы повторно читает и реконструирует состояние из файлов. ([GitHub][5])

Это создаёт проблемы:

* отсутствуют транзакции;
* невозможно атомарно обновить несколько частей состояния;
* параллельные запросы могут перезаписать друг друга;
* формат хранения становится частью API между слоями;
* трудно мигрировать формат сессии;
* невозможно нормально контролировать жизненный цикл сессий.

SQLite здесь уже был бы надёжнее набора JSON-файлов. Сами CSV можно оставить файлами, но метаданные и состояние обработки лучше хранить структурированно.

### 5. Несколько мест повторно реализуют одни и те же вычисления

Обработка, построение исходного графика и формирование итогового графика отдельно вычисляют производные, сглаживание и пики. Из-за этого график исходных данных может использовать не совсем тот же алгоритм, что итоговая обработка. ([GitHub][3])

Должен существовать один вычислительный pipeline:

```text
Parse
  → Validate
  → Normalize units
  → Align axes
  → Process
  → Detect peaks
  → Calculate effects
  → Produce immutable result
```

Графики должны только отображать готовый результат, а не повторно выполнять часть научного расчёта.

### 6. Отсутствуют порты между прикладным слоем и инфраструктурой

Например, use case получает конкретный `SessionStorage`, а не интерфейс вида:

```python
class SessionRepository(Protocol):
    def load(self, session_id: SessionId) -> Session: ...
    def save(self, session: Session) -> None: ...
```

Аналогично стоит выделить:

* `ThermogramParser`;
* `SessionRepository`;
* `ArchiveService`;
* `ProcessingEngine`;
* `PlotPresenter`.

Это не обязательно должно быть «чистой архитектурой» во всех деталях, но вычислительная логика не должна знать, что сессия хранится именно в каталоге с JSON и CSV.

# Проблемы кроме архитектуры

## Критическая уязвимость: обход каталогов через cookie сессии

Это самая опасная проблема, которую я увидел.

Идентификатор сессии берётся из cookie. Затем `session_dir()` заменяет только символ `/` на `_`, но не запрещает значение `..` и не проверяет, что итоговый путь остаётся внутри корневого каталога сессий. Метод проверки сессии принимает каталог, если он просто существует. ([GitHub][6])

Цепочка может выглядеть так:

```text
Cookie: tg_session=..
session_root / ".."
→ родительский каталог session_root
→ каталог существует
→ приложение принимает его за действующую сессию
```

Особенно опасен экспорт сессии: архиватор рекурсивно упаковывает каталог сессии. При таком cookie он потенциально может упаковать родительский каталог, то есть файлы, которые вообще не принадлежат пользовательской сессии. ([GitHub][7])

Исправление должно включать сразу несколько мер:

```python
SESSION_ID_RE = re.compile(r"^[0-9a-f]{32}$")

def session_dir(session_id: str) -> Path:
    if not SESSION_ID_RE.fullmatch(session_id):
        raise InvalidSessionId()

    path = (root / session_id).resolve()
    path.relative_to(root.resolve())
    return path
```

Кроме того, идентификатор cookie должен быть подписан либо сопоставляться с серверной записью, а не использоваться напрямую как имя каталога.

## Незащищённая cookie сессии

Cookie устанавливается с `HttpOnly` и `SameSite=Lax`, но без подписи, `Secure`, срока жизни и ротации. Зная идентификатор другой сессии, клиент потенциально может запросить её файлы. ([GitHub][6])

Для публичного сервиса это фактически проблема контроля доступа, даже если полноценной авторизации пока нет.

## Файлы полностью читаются в память без лимитов

Upload обрабатывается через `await upload.read()`, затем содержимое кодируется в Base64, а парсер снова декодирует его. Получаются несколько полных копий файла в памяти. Ограничений на размер файла, количество строк, число файлов и расширение не видно. ([GitHub][8])

Большой файл может вызвать:

* резкий рост RAM;
* блокировку event loop;
* длительную загрузку CPU;
* заполнение диска;
* отказ в обслуживании.

Base64 между web-слоем и прикладным слоем здесь вообще не нужен. Лучше передавать временный файл или поток.

## Небезопасный импорт архивов сессий

Архив распаковывается напрямую в каталог текущей сессии. Нет ограничений:

* на общий распакованный размер;
* число файлов;
* степень сжатия;
* допустимые имена файлов;
* структуру архива;
* допустимые значения импортированных настроек. ([GitHub][7])

Даже без обхода путей ZIP-бомба может занять весь диск или память.

Также импортированная сессия может подставить экстремальные значения параметров обработки, например огромное число точек интерполяции, и вызвать тяжёлый расчёт.

## Архив сессии может включать сам себя

Выходной `.tg` создаётся внутри каталога сессии, после чего весь этот каталог рекурсивно перебирается для добавления файлов в архив. Файл экспорта явно не исключается, поэтому он может попасть в собственный архив во время формирования. ([GitHub][7])

Архив следует создавать во временном каталоге вне экспортируемой директории и затем отдавать клиенту.

# Возможные ошибки научных расчётов

## Усреднение выполняется по номеру строки, а не по физической оси

Термограммы приводятся к одинаковому количеству строк через нормализованный индекс от 0 до 1, а затем усредняются по соответствующим строкам. Температурная или временная координата для выравнивания не используется. ([raw.githubusercontent.com][9])

Например:

* эксперимент A: 20–500 °C;
* эксперимент B: 100–800 °C.

После текущей нормализации середина первого эксперимента будет усреднена с серединой второго, хотя физические температуры там различаются.

Это может сделать недостоверными:

* среднюю кривую;
* производную;
* положение пиков;
* площадь эффектов;
* сравнение экспериментов.

Нужно сначала выбрать общую физическую сетку — обычно температуру или время — а затем интерполировать каждый эксперимент именно на неё.

## Пропущенные данные молча заменяются придуманными значениями

После преобразования данных вызываются `ffill()`, `bfill()` и затем `fillna(0.0)`. Таким образом пропуски могут заменяться предыдущим значением, следующим значением или нулём. ([GitHub][9])

Для научных данных это опасно: программа не сообщает, что данные повреждены, а создаёт синтетические точки, которые затем участвуют в производной и поиске пиков.

Лучше:

* отклонять критические пропуски;
* интерполировать только по явно выбранной физической оси;
* сохранять маску интерполированных точек;
* предупреждать пользователя о доле восстановленных данных.

## Парсер может незаметно повреждать CSV

Парсер последовательно пробует разные разделители, перехватывает широкие исключения и в ряде случаев возвращает пустой DataFrame вместо диагностической ошибки. Числовые ошибки затем преобразуются в `NaN`, а позднее могут превратиться в нули. ([GitHub][10])

Кроме того, для CSV без заголовка стандартное определение заголовка pandas может принять первую строку данных за имена колонок. В результате первая точка эксперимента будет потеряна.

Нужен отдельный этап определения формата с отчётом:

```text
encoding
separator
header detected
column mapping
row count
invalid numeric cells
temperature range
time range
```

## Скорость нагрева вычисляется сомнительным способом

В итоговой сводке скорость нагрева определяется как отношение последней температуры к последнему времени. Это корректно только в частном случае, когда начальные температура и время равны нулю. ([GitHub][11])

Надёжнее использовать:

```text
(T_last - T_first) / (t_last - t_first)
```

или линейную регрессию температуры по времени на выбранном диапазоне.

## Недокументированный коэффициент расчёта эффекта

В вычислении эффекта применяется фиксированный коэффициент `0.4458333`, но рядом нет единиц измерения, ссылки на методику или объяснения происхождения. ([GitHub][11])

Для научного приложения каждая такая константа должна иметь:

* название;
* физический смысл;
* единицы;
* источник;
* область применимости;
* тест с известным эталонным результатом.

## Результат зависит от установленного окружения

Код напрямую использует NumPy, но NumPy не указан как явная зависимость проекта. SciPy импортируется опционально: если он установлен, используется один алгоритм сглаживания, если отсутствует — собственная fallback-реализация. При этом SciPy также не указан в зависимостях. ([GitHub][12])

В двух окружениях приложение может дать разные результаты на одном файле. Для научного ПО это серьёзная проблема воспроизводимости.

## Сглаживание через fallback может искажать данные

Fallback для фильтра Савицкого — Голея не является полностью эквивалентной заменой SciPy. Кроме того, валидные значения извлекаются из массива подряд, из-за чего участки, разделённые пропусками, могут фактически соединиться перед фильтрацией. ([GitHub][13])

Такой алгоритм обязательно нужно проверять на эталонных кривых, особенно на границах и около пропусков.

# Проблемы производительности и надёжности

### CPU-расчёты выполняются внутри async-обработчиков

Маршрут `/process` объявлен асинхронным, но внутри выполняет синхронные pandas/NumPy-расчёты и файловые операции. Пока идёт обработка одной большой термограммы, тот же worker не сможет нормально обслуживать другие запросы. ([GitHub][14])

Минимальное исправление — запускать обработку через thread pool. Для тяжёлых расчётов лучше отдельный process pool или очередь заданий.

### Нет блокировок и атомарной записи

JSON и CSV записываются прямо в итоговые файлы. Если два запроса одновременно изменят одну сессию или процесс будет прерван во время записи, файл может оказаться частично записанным. ([GitHub][5])

Нужны:

* запись во временный файл;
* `fsync`;
* атомарный `replace`;
* блокировка на уровне сессии;
* версия состояния.

### Нет очистки старых сессий и квот

В файловом хранилище не видно TTL, ограничения общего объёма, лимита на сессию или фоновой очистки. ([GitHub][5])

При эксплуатации диск будет постепенно заполняться исходными файлами, результатами и архивами.

### График исходных данных показывает только первую термограмму

В use case для исходного графика из набора загруженных данных выбирается первый элемент. Остальные исходные термограммы не отображаются через этот путь. ([github.com][3])

Это либо функциональная ошибка, либо поведение, которое нужно явно отражать в интерфейсе.

# Проблемы качества проекта

В просмотренной ветке не видно каталога тестов и CI-конфигурации. При этом проект содержит вычислительные алгоритмы, импорт нескольких форматов, архивирование и управление сессиями — все эти части особенно нуждаются в автоматических тестах. В репозитории также присутствует сгенерированный каталог `src/tgapp.egg-info`, который обычно не должен храниться в Git. ([GitHub][1])

В README остались команды с переходом в каталог `python-app`, хотя Python-проект уже находится в корне рассматриваемой структуры. Это признак того, что документация отстаёт от реорганизации репозитория. ([GitHub][15])

Зависимости имеют только нижние границы версий, lock-файла не видно. Обновление одной транзитивной зависимости может изменить поведение расчётов или сломать запуск. ([GitHub][12])

Часть функций пока явно не реализована, например экспорт графика возвращает HTTP 501. Для каркаса миграции это нормально, но такое приложение пока нельзя считать функционально завершённым. ([GitHub][16])

## Что исправлять в первую очередь

1. **Закрыть обход каталогов:** строгая проверка session ID, проверка принадлежности пути корню, подписанная cookie, экспорт во внешний временный каталог.
2. **Ввести лимиты:** размер upload, число файлов, число строк, размеры ZIP, дисковые квоты и TTL сессий.
3. **Проверить математику:** выравнивание термограмм по температуре или времени, обработка пропусков, скорость нагрева, площади эффектов.
4. **Выбрать один web-стек:** удалить либо изолировать Dash-ветку.
5. **Выделить единый ProcessingEngine:** графики не должны повторно рассчитывать производные и пики.
6. **Разделить прикладной слой и инфраструктуру:** интерфейсы хранилища, парсера и архиватора.
7. **Добавить эталонные тесты:** одинаковые входные данные должны давать результаты, совпадающие с исходным R-приложением в пределах заданной погрешности.
8. **После этого заняться эксплуатацией:** блокировки, атомарные записи, очередь расчётов, lock-файл зависимостей и CI.

Наиболее серьёзные проблемы сейчас — не эстетика архитектуры, а **возможность чтения посторонних каталогов через session cookie** и **вероятно некорректное усреднение экспериментальных кривых по номеру строки вместо физической координаты**. Их стоит устранить до продолжения функциональной миграции.
