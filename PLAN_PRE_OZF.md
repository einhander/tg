# План доработки проекта `tg`

## 1. Цель

Довести текущий конвейер обработки термограмм до следующих свойств:

* существует только один научный pipeline;
* ошибки расчёта не скрываются и не меняют алгоритм;
* DTG воспроизводимо рассчитывается по физической оси времени;
* пропуски обрабатываются по явно заданным правилам;
* температура и время не искажаются сглаживанием;
* температурная коррекция либо применяется полностью, либо расчёт завершается понятной ошибкой;
* результаты содержат достаточно метаданных для проверки метода;
* тесты проверяют физические свойства алгоритмов, а не только отсутствие исключений.

## 2. Вне области задачи

В рамках этой доработки не выполнять:

* добавление OFW/OZF, KAS, Friedman и других кинетических методов;
* настройку CI;
* миграцию файлового хранилища на БД;
* полную замену FastAPI/Jinja-интерфейса;
* крупный редизайн UI;
* изменение исходных научных формул без отдельного обоснования;
* восстановление Dash-кода.

# 3. Этап 1. Удалить автоматический legacy fallback

## Проблема

`process_thermograms()` сначала вызывает новый `ProcessingEngine`, но перехватывает любое исключение через `except Exception` и запускает старый `domain.processing.process_thermograms`. Пользователь получает результат, не зная, что использован другой алгоритм.

## Задачи

1. Удалить импорт:

```python
from tgapp.domain.processing import process_thermograms as _domain_process
```

2. Удалить обе legacy-ветки:

   * обработку при отсутствии validated-файлов;
   * fallback после ошибки `ProcessingEngine`.

3. При отсутствии валидированных термограмм возвращать статус:

```text
no-valid-thermograms
```

с перечнем причин, по которым файлы не прошли проверку.

4. Не использовать `except Exception` для доменных ошибок.

5. Разделить ошибки на категории:

```python
ThermogramValidationError
InvalidProcessingSettingsError
NoCommonRangeError
CorrectionRangeError
ProcessingCalculationError
```

6. Неожиданные программные ошибки:

   * логировать с traceback;
   * не показывать traceback пользователю;
   * возвращать общий статус `internal-processing-error`;
   * не запускать другой алгоритм.

7. После переноса необходимых вспомогательных функций удалить:

```text
src/tgapp/domain/processing.py
```

8. Выполнить поиск оставшихся legacy-вызовов:

```bash
rg -n "domain\.processing|_domain_process|legacy|fallback" src tests
```

## Критерии готовности

* Любая обработка проходит через `ProcessingEngine`.
* В коде нет автоматического выбора второго научного алгоритма.
* Ошибка нового pipeline не создаёт `processed.csv`.
* Пользователь видит понятное сообщение об ошибке.
* В metadata записывается статус неудачного расчёта.

# 5. Этап 3. Исправить обработку NaN и бесконечных значений

## Проблема

Текущий validator сначала применяет `np.isfinite()` и отклоняет массив с любым `NaN`, поэтому расположенный ниже код удаления и интерполяции пропусков фактически недостижим. Тесты закрепляют это поведение.

## Целевая политика

### `inf` и `-inf`

Всегда являются ошибкой во всех физических колонках:

* temperature;
* time;
* mass;
* DTA.

### NaN в независимых осях

Строки с NaN в температуре или времени удаляются.

После удаления необходимо повторно проверить:

* минимальное число точек;
* монотонность;
* температурный диапазон;
* строгое возрастание времени.

### NaN в массе

* начальные и конечные NaN удаляются;
* отдельные внутренние NaN разрешено интерполировать;
* длинные разрывы отклоняются либо требуют явно заданного лимита.

Добавить настройку:

```python
max_interpolation_gap_points: int
max_interpolated_fraction: float
```

### NaN в DTA

Использовать такую же политику, но независимо от массы.

Если DTA отсутствует полностью, TG/DTG могут быть рассчитаны без DTA.

## Рекомендуемый порядок

```text
проверка длины
→ отделение inf от NaN
→ удаление строк с NaN в осях
→ удаление краевых NaN сигналов
→ интерполяция допустимых внутренних разрывов
→ проверка доли восстановленных данных
→ проверка осей
→ создание ValidatedThermogram
```

## Metadata

Сохранять:

```python
{
    "rows_original": ...,
    "rows_removed_axis_nan": ...,
    "rows_removed_edge_nan": ...,
    "mass_points_interpolated": ...,
    "dta_points_interpolated": ...,
    "interpolated_fraction": ...,
}
```

## Тесты

Добавить тесты:

1. Внутренний одиночный NaN массы интерполируется.
2. Начальный NaN массы удаляется.
3. NaN температуры приводит к удалению всей строки.
4. `inf` массы вызывает ошибку.
5. Разрыв длиннее лимита вызывает ошибку.
6. Слишком большая доля восстановленных значений вызывает ошибку.
7. После очистки остаётся меньше минимального числа точек — ошибка.
8. Metadata содержит корректные счётчики.

## Критерии готовности

* Нет противоречия между документацией, кодом и тестами.
* NaN не превращается в ноль.
* Интерполяция применяется только внутри диапазона.
* Пользователь получает предупреждение о восстановленных данных.

# 6. Этап 4. Гарантировать корректные физические оси

## 6.1. Время

Для расчёта:

```python
np.gradient(mass, time)
```

время должно быть строго возрастающим.

Заменить проверку:

```python
np.any(np.diff(time) < 0)
```

на проверку с минимальным разрешённым шагом:

```python
np.any(np.diff(time) <= time_epsilon)
```

Добавить параметр или константу:

```python
TIME_EPSILON = 1e-9
```

Повторяющиеся timestamps не оставлять в данных.

Допустимые варианты:

* объединить соседние точки с одинаковым временем;
* отклонить файл.

Для первой версии предпочтительнее отклонять, поскольку усреднение массы при одном времени может скрыть особенности исходного сигнала.

## 6.2. Температура

`np.interp()` требует упорядоченную ось. Текущий допуск небольшого обратного движения температуры не гарантирует строгую монотонность перед интерполяцией.

Ввести два режима:

### Строгий режим

Любое обратное движение температуры больше `temperature_epsilon` вызывает ошибку.

### Режим очистки приборного шума

Разрешает небольшие локальные колебания, после чего:

1. строится монотонная температурная последовательность;
2. повторяющиеся температуры объединяются;
3. факт коррекции записывается в metadata.

Не применять обычную глобальную сортировку строк, поскольку она может изменить временной порядок эксперимента.

## Тесты

* Повторяющееся время вызывает ошибку.
* Почти одинаковое, но возрастающее время принимается.
* Слабый температурный шум корректируется предсказуемо.
* Существенное обратное движение температуры отклоняется.
* После очистки температура подходит для `np.interp`.
* Время после интерполяции остаётся строго возрастающим.

# 7. Этап 5. Исправить расчёт DTG

## Текущий правильный принцип

DTG уже рассчитывается для каждого опыта отдельно:

```python
dmdt = np.gradient(mass, time)
```

и только затем производные усредняются. Этот порядок сохранить.

## Задачи

1. Перед вызовом `np.gradient` проверять:

   * длину не менее трёх точек;
   * конечность массы и времени;
   * строгое возрастание времени;
   * минимальный шаг времени.

2. Не возвращать фиктивную серию из 500 `NaN` при ошибке входа.

Вместо этого выбрасывать:

```python
DerivativeCalculationError
```

3. Указать параметр `edge_order` явно:

```python
np.gradient(mass, time, edge_order=2)
```

если точек достаточно.

4. В metadata результата сохранять:

```python
{
    "derivative_definition": "dm/dt",
    "derivative_units": "...",
    "gradient_method": "numpy.gradient",
    "gradient_edge_order": 2,
    "per_run_derivative": True,
}
```

5. Не округлять DTG до:

   * поиска пиков;
   * расчёта площади;
   * других научных операций.

Округление выполнять только при экспорте или формировании отображаемой таблицы.

6. Сохранять отдельные DTG-кривые опытов, а не только среднюю:

```python
ProcessingResult.per_run
```

или:

```python
tuple[ProcessedRun, ...]
```

Это понадобится для диагностики и будущего кинетического анализа.

## Аналитические тесты

1. `m(t) = m₀ − at` даёт `dm/dt = −a`.
2. `m(t) = m₀ − at²` даёт `dm/dt = −2at`.
3. Сдвиг времени на константу не меняет DTG.
4. Изменение единицы времени изменяет численное значение производной ожидаемым образом.
5. Разная частота измерений не меняет центральную часть производной сверх допуска.
6. Повтор времени вызывает доменную ошибку.
7. Производная среднего не используется вместо среднего производных.

# 9. Этап 7. Исправить Savitzky–Golay на участках с пропусками

## Проблема

Текущий код собирает все конечные значения в один массив, сглаживает их и возвращает обратно. Точки по разные стороны NaN-разрыва влияют друг на друга.

## Задачи

1. Реализовать выделение непрерывных конечных сегментов:

```python
def finite_segments(values: np.ndarray) -> list[slice]:
    ...
```

2. Каждый сегмент сглаживать отдельно.

3. Для короткого сегмента:

   * не сглаживать;
   * записывать предупреждение;
   * не скрывать `ValueError`.

4. Валидировать параметры до запуска:

   * окно нечётное;
   * `window > polyorder`;
   * `window <= segment_length`;
   * `polyorder >= 1`.

5. Не изменять параметры молча.

Возможны два режима:

```text
strict — неправильные параметры вызывают ошибку
adjust — параметры корректируются, но изменение фиксируется
```

Для научного расчёта по умолчанию использовать `strict`.

6. Явно задать и документировать `mode` Savitzky–Golay.

## Тесты

* Константа остаётся константой.
* Линейный сигнал сохраняется.
* Полином допустимого порядка сохраняется.
* Левая часть NaN-разрыва не зависит от правой.
* Короткий сегмент не вызывает скрытого изменения параметров.
* Чётное окно вызывает понятную ошибку.
* Окно больше массива вызывает понятную ошибку.

1. Не передавать содержимое файла между слоями в Base64.

2. Читать upload потоково во временный файл с контролем размера:

```text
read chunk
→ update size
→ stop when limit exceeded
→ save temporary file
→ parse
→ delete temporary file
```

3. Проверять лимит по фактически прочитанным байтам, а не только по HTTP-заголовку.

4. Ограничить:

* размер одного файла;
* общий размер запроса;
* число файлов;
* число распарсенных строк.

5. Удалять временные файлы через `finally`.

## Критерии готовности

* Файл не существует одновременно в RAM в исходной, Base64 и декодированной формах.
* Превышение лимита останавливает чтение.
* Временные файлы удаляются после ошибки.

# 15. Этап 13. Усилить научные тесты

Количество тестов не является целью. Каждый тест должен проверять конкретный инвариант.

## Обязательные аналитические fixtures

### Линейная масса

```text
m(t) = m₀ − at
```

Проверка постоянной DTG.

### Квадратичная масса

```text
m(t) = m₀ − at²
```

Проверка линейной производной.

### Гауссов пик

Использовать для проверки:

* положения пика;
* устойчивости при изменении bins;
* влияния сглаживания.

### Линейная температурная программа

```text
T(t) = T₀ + βt
```

Проверка скорости нагрева.

### Треугольный DTA-пик

Проверка аналитически известной площади теплового эффекта.

## Заменить слабые тесты

Удалить или переписать тесты вида:

```text
pipeline не упал на inf
пик находится внутри общего диапазона
результат является ProcessingResult
```

Вместо них проверять:

```text
inf отклонён
координата пика совпадает с эталоном
значение производной совпадает с формулой
ошибка коррекции дошла до пользователя
```

## Регрессионные тесты реальных файлов

Для каждого эталонного файла хранить:

```text
контрольную сумму входа
настройки обработки
диапазон температуры
скорость нагрева
координаты основных пиков
несколько контрольных точек TG/DTA/DTG
тепловой эффект на фиксированном интервале
```

Не сравнивать только знак эффекта.

Если R-результат считается эталоном, хранить конкретные численные значения и допуски.

## Тест воспроизводимости

Два последовательных расчёта с одинаковыми:

* файлами;
* настройками;
* версиями зависимостей

должны давать одинаковые массивы в пределах машинного допуска.
