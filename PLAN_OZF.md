# План добавления кинетического анализа термограмм

## 1. Цель

Добавить в приложение отдельный режим **«Кинетический анализ»**, который рассчитывает энергию активации по нескольким термограммам, полученным при разных скоростях нагрева.

Первая версия должна поддерживать:

1. Ozawa–Flynn–Wall, далее `OFW` или `FWO`.
2. Kissinger–Akahira–Sunose, далее `KAS`.
3. Friedman.
4. Сравнение результатов методов на одном графике.

Следующим этапом, не входящим в MVP:

* усовершенствованный метод Вязовкина;
* обычный метод Киссинджера по температуре максимума;
* определение предэкспоненциального множителя;
* выбор кинетической модели;
* прогнозирование времени превращения.

OFW, KAS и Friedman относятся к изоконверсионным методам: для каждой степени превращения `α` определяется отдельное значение кажущейся энергии активации `Eα`. OFW основан на работах Озавы и Флинна—Уолла.

# 2. Архитектурное размещение

## 2.1. Не добавлять методы в обычный `processing_engine`

Существующий конвейер обработки термограмм должен отвечать за:

```text
parse
→ validate
→ normalize
→ smooth
→ calculate TG/DTA/DTG
→ return processed individual run
```

Новая подсистема должна получать **отдельные обработанные эксперименты**, а не усреднённую термограмму.

Запрещено использовать для кинетики:

```text
ProcessingResult.mean_frame
```

или любой другой результат, в котором несколько скоростей нагрева уже усреднены.

Перед началом работы определить стабильный контракт между обычной обработкой и кинетическим анализом.

Кинетический модуль должен получать объект примерно следующего вида:

```python
@dataclass(frozen=True, slots=True)
class ProcessedRun:
    run_id: str
    source_name: str

    time_s: np.ndarray
```

Не импортировать внутренние классы конкретного `processing_engine`, если они ещё изменяются. Создать отдельный стабильный DTO или protocol.

Новая ветка должна начинаться от коммита, где уже реализованы:

* строгая валидация термограмм;

# 4. Доменная модель кинетического исследования

## 4.1. `KineticRun`

```python
@dataclass(frozen=True, slots=True)
class KineticRun:
    run_id: str
    source_name: str

    temperature_k: np.ndarray
    time_s: np.ndarray
    mass: np.ndarray

    nominal_heating_rate_k_min: float | None
    measured_heating_rate_k_min: float

    initial_mass: float
    final_mass: float

    atmosphere: str | None
    sample_name: str | None
    metadata: Mapping[str, object]
```

Инварианты:

* температура хранится в K;
* время хранится в s;
* температура и время монотонны;
* длины массивов совпадают;
* начальная масса больше конечной для процесса потери массы;
* скорость нагрева положительна;
* все значения конечны;
* опыт не является усреднением нескольких файлов.

## 4.2. `KineticStudy`

```python
@dataclass(frozen=True, slots=True)
class KineticStudy:
    study_id: str
    name: str
    runs: tuple[KineticRun, ...]
    conversion_settings: ConversionSettings
```

Исследование объединяет эксперименты:

* одного материала;
* одного типа процесса;
* в сопоставимой атмосфере;
* с разными скоростями нагрева;
* с одинаково определённой степенью превращения.

Минимум для расчёта — три валидных эксперимента с различными скоростями нагрева. ICTAC прямо рекомендует данные не менее чем от трёх температурных программ.

В интерфейсе рекомендуется предупреждение при трёх опытах и нормальный статус при четырёх и более.

## 4.3. `ConversionSettings`

```python
@dataclass(frozen=True, slots=True)
class ConversionSettings:
    alpha_min: float = 0.05
    alpha_max: float = 0.95
    alpha_step: float = 0.05

    initial_mass_mode: str = "selected_plateau"
    final_mass_mode: str = "selected_plateau"

    initial_range_k: tuple[float, float] | None = None
    final_range_k: tuple[float, float] | None = None

    reaction_range_k: tuple[float, float] | None = None
```

Не зашивать определение начальной и конечной массы внутрь конкретного метода.

# 5. Расчёт степени превращения

Использовать:

```text
α(T) = (m₀ − m(T)) / (m₀ − mf)
```

где:

* `m₀` — начальная масса выбранной стадии;
* `mf` — конечная масса выбранной стадии;
* `α = 0` — начало превращения;
* `α = 1` — завершение превращения.

## 5.1. Определение `m₀` и `mf`

Поддержать два режима:

### Ручной режим

Пользователь выделяет:

* стабильный диапазон до начала разложения;
* стабильный диапазон после завершения выбранной стадии.

Масса определяется как медиана значений на выбранном участке.

### Автоматический режим

Программа предлагает диапазоны плато, но пользователь должен увидеть их на графике и подтвердить.

Автоматически найденные значения нельзя применять скрыто.

## 5.2. Многостадийное разложение

В первой версии анализировать одну выбранную стадию.

Пользователь должен задавать диапазон реакции:

```text
T_start … T_end
```

Степень превращения нормализуется внутри выбранной стадии.

Не применять единственную глобальную `α` к кривой с несколькими перекрывающимися стадиями без предупреждения. ASTM E1641 описывает применение OFW прежде всего к гладким профилям разложения с одним максимумом скорости и использует предположение о кинетике первого порядка.

Не называть реализацию «соответствующей ASTM E1641», пока она не проверена по полному тексту стандарта и его требованиям. Использовать название:

```text
Ozawa–Flynn–Wall isoconversional analysis
```

# 6. Построение общей сетки степеней превращения

Для каждого опыта:

1. Рассчитать `α(T)`.
2. Проверить монотонность `α`.
3. Удалить или объединить повторяющиеся значения `α`.
4. Не сортировать сильно немонотонную кривую молча.
5. Для каждой точки общей сетки `αj` интерполировать:

   * `Tα`;
   * `tα`;
   * для Friedman — `dα/dt`.

Пример сетки:

```text
0.05, 0.10, 0.15, …, 0.90, 0.95
```

Для каждой `αj` использовать только те опыты, которые реально покрывают эту степень превращения.

Не выполнять экстраполяцию.

Если конкретная `αj` доступна менее чем для трёх скоростей нагрева:

* не рассчитывать `Eα`;
* сохранить значение `NaN`;
* добавить диагностическое сообщение.

# 9. Реализация OFW

Для каждой степени превращения `α` построить линейную зависимость:

```text
log10(β) от 1/Tα
```

Энергию активации определить из наклона согласно выбранной и документированной аппроксимации температурного интеграла.

Требования:

1. Использовать температуру только в K.
2. Использовать одну согласованную единицу скорости нагрева.
3. Не копировать коэффициенты из случайных публикаций.
4. В коде дать ссылку на исходную формулу.
5. Константы вынести в именованные значения.
6. Не округлять `Tα` до регрессии.
7. Рассчитывать:

   * slope;
   * intercept;
   * `R²`;
   * стандартную ошибку slope;
   * `Eα`;
   * число использованных опытов.

Пример файла:

```text
domain/kinetics/methods/ofw.py
```

Тестировать отдельно саму регрессионную формулу и полный конвейер.

# 10. Реализация KAS

Для каждой `α` построить:

```text
ln(β / Tα²) от 1/Tα
```

Из наклона определить:

```text
Eα = −slope · R
```

при выбранном направлении оси `1/T`.

KAS является интегральным изоконверсионным методом для динамических экспериментов с различными постоянными скоростями нагрева.

Требования аналогичны OFW:

* K;
* согласованные единицы `β`;
* полный результат регрессии;
* отсутствие округления;
* диагностика недостаточного числа точек.

OFW и KAS должны использовать один модуль линейной регрессии.

# 11. Реализация Friedman

Для каждой `α` построить:

```text
ln(dα/dt) от 1/Tα
```

и определить:

```text
Eα = −slope · R
```

Friedman — дифференциальный изоконверсионный метод, основанный на скоростях превращения. Исходная работа опубликована H. L. Friedman в 1964 году.

## Особые требования

Метод чувствителен к шуму, поэтому:

1. Производную считать для каждого опыта отдельно.
2. Не брать производную усреднённой кривой.
3. Сглаживать `α(t)` до дифференцирования.
4. Настройки сглаживания Friedman хранить отдельно от визуального сглаживания графика.
5. Не допускать `dα/dt <= 0` в логарифм.
6. Показывать число исключённых точек.
7. Предупреждать, если результат сильно зависит от окна сглаживания.

Добавить анализ устойчивости:

```text
расчёт с основным окном
расчёт с окном − 2
расчёт с окном + 2
```

Если изменение `Eα` превышает заданный порог, пометить точку как неустойчивую.

# 12. Общая диагностика качества данных

Перед запуском любого метода проверять:

## 12.1. Количество опытов

* меньше 3 — ошибка;
* 3 — предупреждение о слабой статистике;
* 4 и более — нормальный режим.

## 12.2. Скорости нагрева

* скорости должны различаться;
* одинаковые или почти одинаковые скорости не считать независимыми температурными программами;
* фактическую скорость определять регрессией `T(t)`;
* номинальную скорость сохранять отдельно;
* сравнивать номинальную и измеренную скорость.

## 12.3. Линейность нагрева

Для OFW, KAS и начальной реализации Friedman использовать только участки с близким к линейному нагревом.

Сохранять:

```text
βmeasured
R² для T(t)
максимальное отклонение от линейной программы
```

При существенной нелинейности:

* OFW и KAS не запускать;
* объяснить причину;
* в будущем предложить метод Вязовкина, способный учитывать более общие температурные программы.

## 12.4. Сопоставимость опытов

Проверять и показывать пользователю:

* название материала;
* атмосферу;
* массу образца;
* диапазон температуры;
* направление изменения массы;
* число стадий;
* положение DTG-максимума;
* остаточную массу.

Программа не должна автоматически объединять очевидно разные материалы в одно исследование.

## 12.5. Качество регрессии

Для каждой `α` сохранять:

* `R²`;
* стандартную ошибку;
* число опытов;
* диапазон температур;
* residuals;
* leverage отдельных опытов.

Не удалять выброс автоматически. Разрешить пользователю исключить опыт с обязательной фиксацией причины.

При нажатии на точку `Eα` показывать:

* температуры всех опытов;
* скорости нагрева;
* регрессионную прямую;
* `R²`;
* стандартную ошибку;
* исключённые точки.

# 18. Научные тесты

## 18.1. Генератор синтетических данных

Создать:

```text
tests/fixtures/kinetics/generate_synthetic_runs.py
```

Численно решать уравнение:

```text
dα/dt = A · exp(−E / RT) · f(α)
T(t) = T₀ + βt
```

Начать с реакции первого порядка:

```text
f(α) = 1 − α
```

Генерировать минимум четыре опыта, например:

```text
5, 10, 20, 40 K/min
```

Из `α(t)` формировать массу:

```text
m(t) = m₀ − α(t)(m₀ − mf)
```

## 18.2. Базовые тесты

Для OFW, KAS и Friedman проверить:

* восстановление известной `E`;
* отсутствие зависимости от единицы массы;
* отсутствие зависимости от начала отсчёта времени;
* работу при разной частоте дискретизации;
* работу при разной длине файлов;
* неизменность результата при перестановке опытов;
* отсутствие экстраполяции;
* отказ при двух опытах;
* отказ при одинаковых скоростях нагрева;
* отказ при температуре в °C, ошибочно принятой за K;
* корректную обработку неполного диапазона `α`.

## 18.3. Тесты с шумом

Добавить:

* шум температуры;
* шум массы;
* дрейф базовой линии;
* редкие пропуски;
* разную частоту измерения;
* небольшое отличие фактической скорости от номинальной.

Проверить:

* устойчивость OFW и KAS;
* чувствительность Friedman;
* срабатывание диагностических предупреждений.

## 18.4. Многостадийные данные

Создать синтетическую сумму двух реакций.

Проверить:

* глобальный анализ выдаёт предупреждение;
* анализ отдельно выбранной стадии работает;
* программа не выдаёт одно скалярное значение как универсальную энергию активации.

## 18.5. Регрессионные тесты

Сохранить эталонные результаты синтетических наборов:

```text
tests/fixtures/kinetics/reference/
```

Каждый fixture должен содержать:

* параметры генерации;
* seed;
* истинные `E` и `A`;
* температурные программы;
* ожидаемые результаты;
* допуски.

Допуски задавать отдельно для каждого метода и обосновать в комментариях.

# 19. Проверка формул

Перед объединением кода выполнить независимую проверку:

1. Сравнить реализацию OFW с исходными формулами Озавы и Флинна—Уолла.
2. Сравнить KAS с опубликованной формой линейной зависимости.
3. Сравнить Friedman с исходной работой и рекомендациями ICTAC.
4. Проверить знак slope и перевод `J/mol ↔ kJ/mol`.
5. Проверить использование `ln` и `log10`.
6. Проверить единицы `β`.
7. Проверить температуру в K.
8. Проверить газовую постоянную:

```text
R = 8.31446261815324 J/(mol·K)
```

Добавить unit-тест с вручную рассчитанным небольшим набором регрессионных точек.

# 22. Definition of Done

Функция считается завершённой, когда:

* кинетика находится в отдельном модуле `domain/kinetics`;
* обычная обработка термограмм не зависит от кинетического модуля;
* кинетика получает отдельные опыты, а не усреднённую кривую;
* исследование содержит минимум три различные скорости нагрева;
* температура внутри алгоритмов хранится в K;
* степень превращения рассчитывается документированной формулой;
* пользователь видит и подтверждает `m₀`, `mf` и диапазон стадии;
* интерполяция выполняется по `α`;
* экстраполяция запрещена;
* реализованы OFW, KAS и Friedman;
* каждый метод возвращает `Eα`, slope, intercept, `R²` и ошибку;
* доступно сравнение методов;
* Friedman использует производную отдельного опыта;
* настройки научного сглаживания отделены от визуальных настроек;
* многостадийные данные вызывают предупреждение;
* исходные данные и параметры расчёта сохраняются;
* результат воспроизводим по сохранённому проекту;
* алгоритмы проверены на синтетических данных с известной энергией активации;
* UI показывает регрессию для выбранной степени превращения;
* экспорт содержит полную зависимость `E(α)`, а не только среднее значение;
* приложение не заявляет соответствие ASTM E1641 без отдельной валидации;
* все unit-, integration- и regression-тесты проходят.
