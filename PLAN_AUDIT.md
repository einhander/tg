
# План исправления приложения `tg`

## 1. Цель работы

Привести приложение к единой архитектуре на базе **FastAPI + Jinja2 + JavaScript/HTMX**, полностью удалить неиспользуемую Dash-ветку и исправить ошибки в обработке научных данных.

Основные результаты:

1. В проекте не остаётся Dash-кода и зависимости `dash`.
2. Вычислительный модуль корректно обрабатывает несколько термограмм на общей физической оси.
3. Пропуски и повреждённые данные не заменяются молча нулями.
4. Производные, скорость нагрева, температурная коррекция и тепловой эффект рассчитываются по документированной методике.
5. Научные результаты покрыты аналитическими и регрессионными тестами.
6. Web-слой не выполняет научные расчёты и не знает внутреннее устройство файлового хранилища.
7. Устраняются критические проблемы безопасности сессий и архивов.

---

# 2. Общие ограничения

Агент должен соблюдать следующие правила:

* Не переделывать внешний вид интерфейса без необходимости.
* Не менять пользовательский workflow: загрузка → обработка → график → выделение диапазона → экспорт.
* Не настраивать коэффициенты вручную только ради совпадения с текущим Python-результатом.
* Не использовать текущую Python-реализацию как научный эталон.
* Для проверки поведения использовать:

  1. реальные файлы из `samples/`.
* Не выполнять экстраполяцию за диапазон экспериментальных данных.
* Не применять `fillna(0)` к научным данным.
* Не скрывать ошибки входных данных. Пользователь должен получать понятное сообщение.
* Не оставлять альтернативные алгоритмы, зависящие от того, установлен ли SciPy.
* Каждый этап оформлять отдельным логическим коммитом.
* После каждого этапа запускать тесты и smoke-проверку приложения.

# 6. Этап 3. Ввести строгую модель входной термограммы

Разделить три состояния данных:

```text
ParsedThermogram
ValidatedThermogram
AlignedThermogram
```

Можно использовать dataclass и NumPy/pandas, но объект должен гарантировать:

* одинаковую длину массивов;
* числовые значения;
* наличие температуры, времени и массы;
* монотонную физическую ось;
* известные единицы;
* отсутствие бесконечных значений;
* отсутствие молча подставленных нулей.

Добавить доменные исключения:

```text
ThermogramValidationError
NonMonotonicAxisError
InsufficientDataError
NoCommonRangeError
CorrectionRangeError
InvalidProcessingSettingsError
```

## 6.1. Исправить нормализацию

Удалить цепочку:

```python
ffill().bfill().fillna(0.0)
```

Новая политика:

1. Полностью пустые строки удаляются.
2. Строки без валидной температуры или времени удаляются с предупреждением либо вызывают ошибку при значительном количестве.
3. Внутренние пропуски зависимых величин разрешено интерполировать только внутри существующего диапазона.
4. Начальные и конечные пропуски не экстраполируются.
5. Доля отброшенных и интерполированных строк сохраняется в metadata.
6. Если после очистки осталось меньше минимального числа точек, обработка прекращается.
7. `NaN`, `inf` и `-inf` не должны попадать в результат без явного предупреждения.

## 6.2. Обработка порядка точек

* Проверить монотонность времени.
* Проверить направление изменения температуры.
* Небольшие повторы температуры разрешается объединять детерминированно, например усреднением зависимых величин.
* Существенное обратное движение температуры должно приводить к ошибке или выделению отдельных сегментов, но не к молчаливой сортировке всего эксперимента.

## Критерий завершения

* Пропуск массы не превращается в нулевую массу.
* Повреждённый файл выдаёт диагностическое сообщение.
* В metadata сохраняются сведения об очистке.
* Парсер и научная валидация тестируются отдельно.

# 7. Этап 4. Исправить выравнивание нескольких термограмм

Текущий подход с интерполяцией по нормализованному индексу строки удалить.

## 7.1. Целевая физическая ось

Для стандартной обработки TGA использовать температуру как общую ось:

```text
Tmin_common = max(Tmin каждого эксперимента)
Tmax_common = min(Tmax каждого эксперимента)
temperature_grid = linspace(Tmin_common, Tmax_common, bins)
```

Причины:

* TG, DTA и DTG сравниваются и отображаются по температуре;
* положение эффектов определяется температурой;
* точки с одинаковым относительным номером строки не обязательно соответствуют одинаковой температуре.

Не выполнять экстраполяцию.

Если общий диапазон отсутствует или слишком мал, вернуть `NoCommonRangeError`.

## 7.2. Интерполяция каждого эксперимента

Для каждой термограммы независимо интерполировать на общую температурную сетку:

* массу;
* DTA/`deltatemp`;
* время.

Использовать только валидные точки внутри диапазона конкретного эксперимента.

## 7.3. Режим одного файла

Для одной термограммы использовать её собственный температурный диапазон и сетку из `bins` точек.

## 7.4. Тесты выравнивания

Добавить синтетические тесты:

1. Одна и та же функция, записанная с 100 и 1000 точками, после выравнивания даёт одинаковую кривую.
2. Неравномерная частота измерений не смещает пики.
3. Разные начальные и конечные температуры обрабатываются только на общем диапазоне.
4. Данные за пределами общего диапазона не экстраполируются.
5. Два файла без общего диапазона вызывают понятную ошибку.
6. Перестановка порядка загруженных файлов не меняет результат.

# 8. Этап 5. Исправить расчёт производной DTG

Не рассчитывать DTG только как производную уже усреднённой массы.

Правильный порядок:

```text
валидация отдельного эксперимента
→ выравнивание
→ сглаживание массы отдельного эксперимента
→ расчёт dm/dt отдельного эксперимента
→ усреднение производных
```

Причина: производная среднего и среднее производных совпадают только при определённых условиях, а различающиеся временные сетки и скорости нагрева нарушают эти условия.

## 8.1. Формула

Базовый вариант:

```python
dmdt = np.gradient(mass_smoothed, time)
```

Перед расчётом:

* время должно быть строго возрастающим;
* одинаковые значения времени запрещены;
* единицы времени должны быть известны;
* знак производной сохраняется и документируется.

## 8.2. Параметр `difflag`

Исследовать использование `difflag` в R-приложении.

Допустимые решения:

* реализовать центральную разность с физически определённым интервалом;
* заменить параметр на размер окна производной;
* объявить устаревшим, если он больше не имеет физического смысла.

Не пересчитывать `difflag` через число bins и максимальное значение времени без документированного вывода формулы.

## 8.3. Аналитические тесты

* Для `m(t) = m0 - a·t` должно получаться `dm/dt = -a`.
* Для квадратичной массы производная должна совпадать с аналитической в пределах допуска.
* Изменение количества bins не должно существенно менять интегральную форму DTG.
* Два эксперимента с разной частотой измерений, но одинаковой функцией массы, должны давать одинаковую DTG-кривую.

# 9. Этап 6. Сделать сглаживание воспроизводимым

Использовать SciPy как обязательную зависимость.

Удалить runtime-ветвление:

```text
SciPy установлен → один алгоритм
SciPy отсутствует → другой алгоритм
```

## Требования

1. Использовать `scipy.signal.savgol_filter`.
2. Проверять:

   * окно нечётное;
   * окно больше `polyorder`;
   * окно не больше длины непрерывного сегмента.
3. Обрабатывать каждый непрерывный участок данных отдельно.
4. Не соединять значения по разные стороны `NaN`-разрыва.
5. Явно выбрать и документировать `mode` обработки краёв.
6. Одинаковые входные данные должны давать одинаковый результат в любом окружении.

## Тесты

* Константа после сглаживания остаётся константой.
* Линейная функция не искажается.
* Полиномиальная функция допустимого порядка сохраняется.
* Разрыв с `NaN` не вызывает взаимного влияния левой и правой частей.
* Некорректные параметры не исправляются молча, а нормализуются с предупреждением либо отклоняются.

# 10. Этап 7. Исправить температурную коррекцию

Корректирующая кривая должна интерполироваться по температуре, а не по относительному номеру строки.

Порядок:

```text
валидация correction-файла
→ проверка температурного диапазона
→ интерполяция correction.deltatemp на temperature_grid
→ применение к каждой кривой или к средней кривой согласно R-методике
```

Не выполнять экстраполяцию.

Если коррекция не покрывает рабочий диапазон:

* вернуть понятную ошибку;
* указать фактический и требуемый диапазоны.

Добавить тест, где коррекция и термограмма имеют разное количество строк, но одинаковую физическую температурную зависимость.

# 11. Этап 8. Исправить скорость нагрева

Удалить расчёт:

```python
last_temperature / last_time
```

Минимально корректная формула:

```text
β = (T_last - T_first) / (t_last - t_first)
```

Предпочтительный вариант для реальных шумных данных — коэффициент линейной регрессии:

```text
T(t) = T0 + βt
```

Дополнительно вычислять:

* `β`, K/мин;
* коэффициент качества линейного приближения;
* минимальную и максимальную локальную скорость нагрева при необходимости.

Если нагрев существенно нелинеен, интерфейс должен показывать, что значение является средней скоростью.

## Тесты

* `T(t) = 25 + 10t` → `10 K/мин`.
* Начальная температура, отличная от нуля, не влияет на скорость.
* Сдвиг начала времени не влияет на скорость.
* Нелинейная программа нагрева помечается как нелинейная.

# 12. Этап 9. Исправить расчёт теплового эффекта

Вынести расчёт из `summary.py` в отдельный доменный модуль:

```text
src/tgapp/domain/effects.py
```

## Алгоритм

1. Нормализовать границы `xmin`, `xmax`, не округляя их до целых градусов.
2. Отсортировать данные по температуре.
3. Интерполировать значения DTA точно на обеих границах.
4. Добавить граничные точки в выбранный интервал.
5. Построить линейную базовую линию между значениями сигнала на границах.
6. Интегрировать:

```text
integral(DTA(T) - baseline(T)) dT
```

7. Применить калибровочный коэффициент только после документирования его смысла.
8. Разделить на начальную массу с проверкой:

   * масса положительна;
   * единицы массы известны;
   * итоговые единицы действительно соответствуют Дж/г.

## Тесты

* Чистая линейная базовая линия даёт нулевой эффект.
* Треугольный пик даёт аналитически известную площадь.
* Результат почти не зависит от числа bins.
* Перестановка границ интервала не меняет результат.
* Выделение границ между существующими точками работает за счёт интерполяции.
* Нулевая и отрицательная масса отклоняются.
* Интервал вне диапазона данных отклоняется.

# 13. Этап 10. Проверить поиск пиков

Использовать один воспроизводимый алгоритм, желательно на базе:

```python
scipy.signal.find_peaks
```

Для минимумов применять поиск пиков к отрицательному сигналу.

Параметры должны иметь понятный смысл:

* prominence;
* минимальное расстояние по температуре;
* минимальная ширина;
* допустимый температурный диапазон.

Не вычислять минимальное расстояние через среднее расстояние между уже найденными пиками.

## Тесты

* Один синтетический пик находится в известной температуре.
* Два близких пика разделяются или объединяются согласно настройкам.
* Небольшой шум не создаёт десятки ложных пиков.
* Минимумы и максимумы не смешиваются.
* Положение пика устойчиво при изменении частоты дискретизации.
* Пики определяются на неокруглённых данных.

Сравнить результат с R, но все намеренные отличия описать в `docs/scientific-method.md`.

# 14. Этап 11. Сформировать единый вычислительный конвейер

Создать один доменный сервис, например:

Целевой pipeline:

```text
parse
→ validate
→ normalize units
→ determine common physical range
→ align each experiment
→ smooth each experiment
→ calculate derivatives for each experiment
→ apply correction
→ aggregate traces
→ detect peaks
→ calculate summary
→ return immutable ProcessingResult
```

Графики не должны:

* повторно сглаживать данные;
* повторно рассчитывать производные;
* повторно искать пики;
* изменять научные значения.

Plotting получает готовый `ProcessingResult` и только преобразует его в Plotly figure.

Функция `get_plot_payload()` не должна повторно вызывать `detect_peaks()` для уже обработанного файла.

# 15. Этап 12. Исправить границы архитектурных слоёв

## 15.1. Целевая зависимость

```text
web → application → domain
                 ↓
              ports
                 ↑
          infrastructure
```

Domain не импортирует:

* FastAPI;
* Plotly;
* файловое хранилище;
* Jinja2;
* web DTO.

Application не импортирует конкретные инфраструктурные реализации.

## 15.2. Создать порты

Создать:

```text
src/tgapp/application/ports.py
```

Минимальные интерфейсы:

```python
class SessionRepository(Protocol):
    ...

class ThermogramParser(Protocol):
    ...

class SessionArchiveService(Protocol):
    ...

class ProcessingResultRepository(Protocol):
    ...
```

Конкретные реализации оставить в `infrastructure/`.

## 15.3. Разделить use cases

Вместо одного большого `use_cases.py` создать:

```text
application/use_cases/
  create_session.py
  upload_thermograms.py
  upload_correction.py
  process_thermograms.py
  calculate_effect.py
  import_session.py
  export_session.py
```

Use case должен:

* принимать DTO и интерфейсы;
* вызывать domain;
* сохранять результат через repository;
* возвращать DTO.

Он не должен строить Plotly JSON или вручную работать с путями.

## 15.4. Composition root

Создание конкретных объектов выполнять в:

```text
src/tgapp/web/app.py
```

или отдельном:

```text
src/tgapp/bootstrap.py
```

Там соединяются:

* FastAPI;
* файловый repository;
* parser;
* archive service;
* processing engine.

# 16. Этап 13. Закрыть критические проблемы безопасности

Эти исправления обязательны, даже если основная задача посвящена архитектуре и расчётам.

## 16.1. Безопасный session ID

* Использовать UUID или строгий hex-идентификатор.
* Валидировать cookie до обращения к файловой системе.
* После `resolve()` проверять, что путь находится внутри `APP_SESSION_DIR`.
* Не заменять только `/`; значения `..`, обратные слеши и абсолютные пути должны отклоняться.
* Подписать cookie либо использовать серверное сопоставление токена и session ID.
* В production устанавливать `Secure`.

## 16.2. Безопасный импорт архива

Перед распаковкой:

* запретить абсолютные пути;
* запретить `..`;
* запретить symlink;
* ограничить количество файлов;
* ограничить суммарный распакованный размер;
* ограничить коэффициент сжатия;
* распаковывать во временный каталог;
* валидировать структуру;
* только затем атомарно импортировать данные.

Не использовать безусловный `ZipFile.extractall()`.

## 16.3. Безопасный экспорт

* Создавать архив вне экспортируемого каталога.
* Не позволять архиву включить самого себя.
* Использовать временный файл.
* Удалять временный файл после отправки.

## 16.4. Ограничения загрузок

Добавить конфигурацию:

```text
APP_MAX_UPLOAD_SIZE
APP_MAX_UPLOAD_FILES
APP_MAX_ARCHIVE_SIZE
APP_MAX_UNPACKED_SIZE
APP_MAX_DATA_ROWS
```

Не читать неограниченный файл целиком в память и не создавать лишнюю Base64-копию.

# 17. Этап 14. Надёжность хранения и выполнения

Файловое хранилище пока можно оставить, но:

* записывать JSON и CSV во временный файл;
* выполнять атомарный `replace`;
* не оставлять частично записанные результаты;
* добавить блокировку на одну сессию;
* не допускать одновременной обработки одной сессии;
* добавить TTL и очистку старых сессий;
* ограничить размер одной сессии.

CPU-bound расчёты не выполнять непосредственно в event loop. Использовать thread pool или process pool через FastAPI/AnyIO.

Миграцию на SQLite в рамках этой задачи не выполнять. Архитектура repository должна позволить сделать её позже без изменения domain и use cases.

# 18. Этап 15. Обновить UI ошибок

Пользователь должен видеть конкретные причины отказа:

* файл не распознан;
* отсутствует колонка;
* недостаточно точек;
* время немонотонно;
* нет общего температурного диапазона;
* correction-файл не покрывает диапазон;
* неверные параметры сглаживания;
* начальная масса неположительна;
* архив повреждён или превышает лимиты.

Не показывать traceback и внутренние пути сервера.

Предупреждения о восстановлении данных должны отображать:

* число удалённых строк;
* число интерполированных точек;
* фактический рабочий диапазон;
* сужение диапазона при усреднении нескольких файлов.
